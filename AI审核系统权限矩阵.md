# AI审核系统权限矩阵

## 1. 设计原则

本项目继续保留 RuoYi 的权限底座，即继续使用 `sys_user`、`sys_role`、`sys_menu`、`sys_user_role`、`sys_role_menu`、`sys_role_dept` 等表，以及后端 `@PreAuthorize("@ss.hasPermi('...')")`、前端 `v-hasPermi` 的权限控制方式。

本次权限整理的目标不是重写权限系统，而是在 RuoYi 的角色、菜单、按钮权限基础上，按 AI 审核业务岗位重新梳理权限边界。

核心原则如下：

1. 最小权限原则：普通用户只拥有完成本岗位工作所需的菜单和按钮权限。
2. 前后端一致：前端控制按钮可见性，后端接口必须做真实权限校验。
3. 菜单权限和数据权限分离：菜单/按钮权限决定“能不能操作”，数据权限决定“能操作哪些数据”。
4. 审核结论权限收紧：审核、退回、终审、删除、重新分析等动作必须明确授权。
5. 默认 `common` 角色不再作为业务通用高权限角色使用，应拆分为更具体的业务岗位角色。

## 2. 岗位角色定义

| 岗位角色 | 建议 role_key | 定位 |
|---|---|---|
| 超级管理员 | `admin` | 平台最高权限，保留 RuoYi 超级管理员能力 |
| 系统管理员 | `system_admin` | 业务层面的系统管理员，拥有全部 AI 审核业务权限，但不管理用户、角色、菜单 |
| 审核员 | `auditor` | 审核系统中提交的全部审核任务，查看资料和 AI 结果，填写审核意见和处理结论 |
| 普通用户/资料提交人员 | `common` | 提交本人待审核任务；退回后重新上传/修改本人资料；上传公共审核依据/常用文件 |
| 知识库管理员 | `library_admin` | 管理审核依据库、常用资源、任务资源、文件归类和版本 |
| AI任务运维 | `ai_operator` | 管理 AI 队列、失败任务重跑、优先级、执行日志，不修改人工审核结论 |
| 只读审计人员 | `audit_viewer` | 只读查看任务、结果、历史和日志，不做业务操作 |

### 2.1 超级管理员落地规则

超级管理员是系统最高权限角色，必须拥有全部菜单、全部按钮、全部接口和全部数据权限。

当前项目保留 RuoYi 默认机制：

1. `user_id = 1` 视为超级管理员用户。
2. 超级管理员登录后，后端权限集合直接包含 `*:*:*`。
3. 超级管理员查询路由时直接返回全部启用菜单。
4. 超级管理员不受 `@DataScope` 数据权限过滤限制。
5. 超级管理员角色 `role_id = 1`、`role_key = admin` 不允许被普通用户修改或删除。

为保证数据库层也清晰表达“超级管理员拥有全部权限”，新增脚本：

`backend/sql/super_admin_full_permissions.sql`

该脚本会：

1. 确保 `role_id = 1` 的超级管理员角色启用，且 `data_scope = 1`。
2. 确保 `user_id = 1` 的超级管理员账号启用。
3. 确保 `user_id = 1` 绑定 `role_id = 1`。
4. 将 `role_id = 1` 显式关联 `sys_menu` 中全部菜单、目录、按钮。

该脚本是幂等脚本，建议在基础菜单和业务菜单初始化完成后执行一次；后续新增菜单或按钮权限后，也可以重复执行。

### 2.2 审核员落地规则

当前业务不单独设置“审核主管”角色，也不存在任务分配或主动领取任务的概念。审核员是唯一的业务审核角色，可以审核所有提交到系统中的审核任务。

审核员的权限边界如下：

1. 可以查看、查询系统中全部审核任务。
2. 可以查看审核详情、版本追溯、审核资料、AI 分析结果。
3. 可以填写审核意见和处理结论。
4. 可以退回资料提交人员重新上传。
5. 可以触发 AI 重新分析。
6. 可以导出审核报告或审核结果，支持批量导出和全量导出。
7. 可以执行“审核资源审核”和“AI任务人工审核”。
8. 可以只读查看审核依据、常用文件资源、任务文件资源。
9. 不能修改审核任务基础信息。
10. 不能删除审核任务、审核资源、AI任务。
11. 不能新增、修改、删除知识库内容。
12. 不能分配审核资源权限。

落地脚本：

`backend/sql/auditor_role_permissions.sql`

该脚本会创建/更新 `role_key = auditor` 的“审核员”角色，并为其授予以下现有权限：

1. 审核列表查看、查询、详情、历史、导出。
2. AI任务查看、查询、详情、导出、人工审核、重新分析。
3. 审核资源查看、查询、详情、导出、批量下载、打包、重新上传、审核。
4. 审核文件库、常用文件资源、任务文件资源的只读查看权限。

当前 `audit:review:edit` 会修改审核任务基础信息，因此不授予审核员。审核员填写审核意见和处理结论，当前通过 `audit:ai:review` 与 `audit:asset:review` 承载。后续如需在“审核列表”中直接处理审核结论，应新增专用权限，例如 `audit:review:approve`、`audit:review:return`，而不是复用 `audit:review:edit`。

### 2.3 系统管理员落地规则

系统管理员是业务层面的最高管理员，拥有审核员的全部权限，并拥有 AI 审核业务模块的全部业务管理权限。

系统管理员不是 RuoYi 超级管理员：

1. 不使用 `role_key = admin`。
2. 不授予 `*:*:*`。
3. 不触发 `user_id = 1` 的超级管理员逻辑。
4. 不授予用户、角色、菜单等 RuoYi 底座管理权限。

系统管理员的权限边界如下：

1. 可以查看和管理全部审核任务。
2. 可以新增、修改、删除、导出审核任务。
3. 可以查看和管理全部 AI 任务。
4. 可以审核 AI 任务、切换 AI 任务状态、提升优先级、触发重新分析、删除 AI 任务。
5. 可以查看和管理全部审核资源。
6. 可以进行审核资源权限分配、批量下载、打包、重新上传、审核、删除。
7. 可以查看和管理审核文件库、常用文件资源、任务文件资源。
8. 可以新增、修改、删除、导出、归类知识库/资源库业务数据。
9. 可以只读查看业务字典。
10. 不能管理用户、角色、菜单。
11. 不能修改系统参数。
12. 默认不开放系统监控、代码生成等平台运维能力。

落地脚本：

`backend/sql/system_admin_role_permissions.sql`

该脚本会创建/更新 `role_key = system_admin` 的“系统管理员”角色，并为其授予：

1. 全部审核列表业务权限。
2. 全部 AI 任务业务权限。
3. 全部审核资源业务权限。
4. 全部审核文件库、常用资源、任务资源管理权限。
5. 业务字典只读入口。

该脚本会重建 `system_admin` 的菜单权限，确保不会遗留用户、角色、菜单等 RuoYi 底座管理权限。

### 2.4 普通用户落地规则

普通用户沿用 RuoYi 默认角色 `role_key = common`，但不再保留 RuoYi 默认高权限配置。

普通用户的权限边界如下：

1. 只能查看自己提交的审核任务列表。
2. 可以新增待审核任务。
3. 只能在任务被退回后重新上传/修改本人资料。
4. 不能查看或维护别人提交的审核任务。
5. 不能审核任务，不能填写审核意见或处理结论。
6. 不能查看 AI 任务队列，不能触发 AI 分析，不能审核 AI 任务。
7. 可以进入审核文件库并上传公共审核依据/常用文件。
8. 不能新增、修改、删除文件夹。
9. 不能编辑、删除、导出、归类常用文件资源。
10. 不拥有用户、角色、菜单、系统参数、监控、代码生成等 RuoYi 底座权限。

落地脚本：

`backend/sql/common_role_permissions.sql`

该脚本会更新 `role_key = common` 的“普通用户”角色，并为其授予：

1. `audit:review:list`
2. `audit:review:query`
3. `audit:review:add`
4. `audit:review:edit`
5. `audit:library:folder:list`
6. `audit:library:folder:query`
7. `audit:library:common:list`
8. `audit:library:common:query`
9. `audit:library:common:add`

其中 `audit:review:edit` 只用于退回后重新上传/修改本人资料，后端服务层会校验任务归属和退回状态。

## 5. 建议新增业务权限

当前已有权限可以覆盖现有页面和接口，但部分业务动作语义不够清楚。建议后续逐步新增以下权限标识，避免继续把不同业务动作混在 `edit`、`changeStatus`、`review` 里。

| 建议新增权限 | 用途 | 建议授予角色 |
|---|---|---|
| `audit:review:submit` | 提交审核任务 | 超级管理员、系统管理员、普通用户 |
| `audit:review:approve` | 审核通过 | 超级管理员、系统管理员、审核员 |
| `audit:review:return` | 退回修改 | 超级管理员、系统管理员、审核员 |
| `audit:asset:upload` | 上传审核资料 | 超级管理员、系统管理员、普通用户 |
| `audit:asset:download` | 单个下载审核资料 | 超级管理员、系统管理员、审核员、知识库管理员、AI任务运维 |
| `audit:ai:retry` | 重跑失败 AI 任务 | 超级管理员、系统管理员、审核员、AI任务运维 |
| `audit:ai:viewLog` | 查看 AI 执行日志 | 超级管理员、系统管理员、审核员、AI任务运维、只读审计人员 |
| `audit:library:version` | 管理知识库文件版本 | 超级管理员、系统管理员、知识库管理员 |

## 6. 数据权限边界

菜单和按钮权限只能解决“能不能进入页面、能不能点击按钮、能不能调用接口”。审核系统还需要业务数据权限，建议按以下规则补充服务层校验。

| 岗位角色 | 数据权限建议 |
|---|---|
| 超级管理员 | 全部业务数据和系统数据 |
| 系统管理员 | 全部 AI 审核业务数据；不可管理用户、角色、菜单等 RuoYi 底座数据 |
| 审核员 | 可处理系统中全部审核任务；可查看全部任务资料、AI 结果、历史记录 |
| 普通用户/资料提交人员 | 只查看和维护自己提交的任务；仅退回后可重新上传/修改本人资料；可上传公共审核依据/常用文件但不能删除、编辑、归类 |
| 知识库管理员 | 管理知识库和资源库全部数据；审核任务默认只读或关联查看 |
| AI任务运维 | 查看和运维 AI 队列、执行日志、失败任务；不能修改人工审核结论 |
| 只读审计人员 | 按授权范围只读查看任务、结果、历史和日志；默认禁止导出敏感资料 |

## 7. 第一阶段落地建议

第一阶段不建议一次性把所有角色和权限都做得很细，可以先落地以下 4 个核心角色：

1. `system_admin`：系统管理员。
2. `auditor`：审核员。
3. `common`：普通用户/资料提交人员。
4. `library_admin`：知识库管理员。

第二阶段再补充：

1. `ai_operator`：AI任务运维。
2. `audit_viewer`：只读审计人员。

第一阶段重点工作：

1. 新增业务角色。
2. 调整 `sys_role_menu`，收窄原 `common` 角色权限。
3. 保留 `admin` 超级管理员不变。
4. 对审核任务、审核资源、AI任务、知识库资源补充业务数据权限校验。
5. 对关键操作增加更明确的权限标识，例如 `submit`、`approve`、`return`、`retry`。

## 8. 当前实现需要注意的问题

1. 当前初始化 SQL 中 `common` 角色权限过宽，需要通过 `backend/sql/common_role_permissions.sql` 收窄为真实业务普通用户角色。
2. 前端 `v-hasPermi` 只能隐藏按钮，不能替代后端权限校验。
3. RuoYi 的 `@DataScope` 主要解决部门维度的数据权限，审核业务还需要按提交人、任务状态、资源归属进行额外校验。
4. 删除、导出、重新分析、状态切换、终审类权限应谨慎授予。
5. 系统管理员拥有全部业务权限，但不拥有用户、角色、菜单等 RuoYi 底座管理权限。
