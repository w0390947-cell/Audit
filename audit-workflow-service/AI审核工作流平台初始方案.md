# AI 审核工作流服务初始方案

## 1. 背景与目标

我们需要搭建一个可复用的 AI 审核工作流能力：外部业务系统通过接口传入“待审核文件”、用户输入参数和上下文信息，工作流服务从“审核标准知识库”中检索相关标准、规则、案例或模板，然后让 AI 基于检索结果对文件进行审核，最终返回结构化、可追踪、可复核的审核结果。

由于类似工作流预计有三到四个，建议不要只做一个固定脚本，而是搭建一个轻量工作流服务。该服务不需要独立前端界面，主要通过 API、内部 SDK 或消息队列内嵌进现有系统。服务先覆盖文件审核场景，后续可以扩展到其他流程，例如材料合规检查、合同条款审核、项目文档完整性检查、报告质量评价等。

## 2. 核心原则

1. 审核结果必须可追溯：每条审核意见应关联引用的知识库片段、规则编号或来源文件。
2. AI 只负责判断和归纳，不直接替代审核依据：系统需要把检索到的依据明确传给模型，并在结果中保留依据。
3. 工作流配置化：不同审核场景应通过流程定义、知识库范围、提示词模板、输出 Schema 来区分，而不是复制多套代码。
4. 集成边界清晰：工作流服务只负责审核编排、检索、AI 判断和结果结构化；文件来源、用户交互、权限控制和结果展示由外部业务系统承接。
5. 人工可复核：审核结果应支持外部系统进行人工确认、驳回、修改和归档。
6. 先做最小可用闭环：优先实现“系统传入文件与参数 -> 检索知识库 -> AI 审核 -> 返回结构化结果”的完整链路。

## 3. 推荐总体架构

```text
外部业务系统
  |
  v
工作流服务接入层
  |-- REST API
  |-- 内部 SDK
  |-- 消息队列消费者
  |-- 回调/事件通知
  |
  v
工作流后端服务
  |-- 工作流编排服务
  |-- 文件解析服务
  |-- 知识库检索服务
  |-- AI 审核服务
  |-- 结果存储服务
  |
  v
存储层
  |-- 原始文件引用或缓存
  |-- 解析文本与切片
  |-- 向量数据库/全文索引
  |-- 审核任务与审核结果
```

## 4. 工作流抽象设计

建议把每个审核流程抽象为一个 `Workflow`，每个工作流由多个节点组成。

### 4.1 工作流定义

一个工作流建议包含以下配置：

| 字段 | 说明 |
| --- | --- |
| workflow_code | 工作流唯一编码 |
| workflow_name | 工作流名称 |
| description | 使用场景说明 |
| input_schema | 输入要求，例如文件 ID、文件 URL、文本内容、业务参数 |
| knowledge_base_scope | 绑定的知识库范围 |
| retrieval_config | 检索策略，例如 topK、相似度阈值、关键词过滤 |
| prompt_template | AI 审核提示词模板 |
| output_schema | 结构化审核结果格式 |
| post_processors | 结果后处理规则，例如风险等级归一化 |
| enabled | 是否启用 |

### 4.2 标准节点

第一阶段建议内置以下节点：

1. 输入接收节点：接收外部系统传入的文件 ID、文件 URL、二进制文件、纯文本或业务参数。
2. 文件解析节点：将 PDF、Word、Excel、图片 OCR 等转为标准文本。
3. 文本切分节点：按章节、页码、表格区域或语义段落切分。
4. 知识库检索节点：从审核标准知识库中检索相关依据。
5. AI 审核节点：基于待审核内容和检索依据生成审核意见。
6. 结果校验节点：校验 AI 输出是否符合 JSON Schema 或结构化格式。
7. 结果存储节点：保存审核任务、引用依据和审核结论。
8. 回调通知节点：将任务状态、审核结果或失败原因通知外部系统。

## 5. 审核任务流程

```text
1. 外部系统调用接口并指定工作流
2. 外部系统传入待审核文件、文本内容或文件引用
3. 工作流服务解析文件内容
4. 工作流服务按章节或语义块拆分待审核内容
5. 工作流服务针对每个内容块检索知识库
6. AI 基于“内容块 + 检索依据 + 审核规则”生成初步审核意见
7. 工作流服务合并、去重、排序审核意见
8. 工作流服务生成结构化审核结果
9. 工作流服务同步返回结果，或异步回调外部系统
```

### 5.1 接口契约草案

第一版建议优先提供 REST API，后续再按现有系统技术栈封装内部 SDK。

| 接口 | 方法 | 作用 |
| --- | --- | --- |
| /api/audit/workflows | GET | 查询可用工作流 |
| /api/audit/tasks | POST | 创建审核任务 |
| /api/audit/tasks/{task_id} | GET | 查询任务状态与结果摘要 |
| /api/audit/tasks/{task_id}/result | GET | 查询完整结构化审核结果 |
| /api/audit/tasks/{task_id}/review | POST | 外部系统回写人工复核结论 |
| /api/audit/knowledge/documents | POST | 导入知识库文档 |
| /api/audit/knowledge/documents/{document_id}/index | POST | 触发知识库文档解析与索引 |

创建审核任务的输入建议同时兼容几种模式：

```json
{
  "workflow_code": "policy_document_audit",
  "biz_id": "PROJECT-20260510-001",
  "callback_url": "https://example.com/audit/callback",
  "input": {
    "file_id": "FILE-001",
    "file_url": "https://example.com/files/FILE-001",
    "text": null,
    "file_name": "待审核文件.pdf",
    "metadata": {
      "business_type": "project",
      "department_id": "D001",
      "submitter_id": "U001"
    }
  }
}
```

接口返回建议分两类：

1. 同步轻量返回：返回 `task_id`、任务状态、是否已完成、简要结论。
2. 异步完整返回：任务完成后通过 `callback_url` 推送完整 JSON 审核结果，外部系统也可以主动查询结果接口。

## 6. 知识库设计

知识库不建议只做向量库，还应保留结构化元数据和原始来源。

### 6.1 知识条目字段

| 字段 | 说明 |
| --- | --- |
| document_id | 来源文件 ID |
| chunk_id | 知识片段 ID |
| title | 标准或制度名称 |
| section | 章节号或条款号 |
| content | 条款内容 |
| source_file | 来源文件 |
| page_no | 页码 |
| rule_code | 规则编号，可为空 |
| category | 分类，例如财务、合同、项目、合规 |
| effective_date | 生效日期 |
| status | 有效、废止、草案 |
| embedding | 向量表示 |

### 6.2 检索策略

建议采用混合检索：

1. 向量检索：处理语义相近但措辞不同的问题。
2. 关键词检索：保证规则编号、专有名词、金额、日期等精确匹配。
3. 元数据过滤：按业务类型、文件类型、制度版本、有效状态过滤。
4. 重排序：对召回结果进行 rerank，提高最终传给模型的依据质量。

## 7. AI 审核输出格式

建议统一输出 JSON。外部业务系统可以按自身需要渲染为表格、详情页、审批单、报告或导出文件。

示例结构：

```json
{
  "task_id": "AUDIT-20260510-001",
  "workflow_code": "policy_document_audit",
  "summary": {
    "overall_result": "需要整改",
    "risk_level": "medium",
    "total_issues": 3
  },
  "issues": [
    {
      "issue_id": "ISSUE-001",
      "title": "缺少必要审批信息",
      "risk_level": "high",
      "location": {
        "file_name": "待审核文件.pdf",
        "page_no": 2,
        "section": "第三章"
      },
      "problem": "文件中未体现项目负责人审批意见。",
      "basis": [
        {
          "source": "项目管理制度.pdf",
          "section": "第 4.2 条",
          "quote": "项目立项材料应包含项目负责人审批意见。",
          "chunk_id": "KB-000123"
        }
      ],
      "suggestion": "补充项目负责人审批意见，并在提交材料中保留审批记录。",
      "confidence": 0.86
    }
  ]
}
```

## 8. 工作流服务功能范围

### 8.1 第一阶段

第一阶段目标是做出可运行闭环：

1. 工作流注册、启停和查询接口。
2. 审核任务创建接口，支持传入文件 ID、文件 URL、二进制文件、纯文本或业务参数。
3. 知识库文档导入接口，支持解析、切片、入库。
4. 审核任务状态查询接口。
5. 自动检索和 AI 审核。
6. 结构化审核结果返回接口。
7. 异步任务回调或事件通知。
8. 审核日志和模型调用记录。

### 8.2 第二阶段

第二阶段增强可维护性和可配置性：

1. 工作流配置文件或配置接口。
2. 提示词模板管理。
3. 输出 Schema 管理。
4. 知识库版本管理。
5. 审核结果修订接口，供外部系统回写人工复核结论。
6. 规则命中率、误报率、人工修改率统计。

### 8.3 第三阶段

第三阶段面向稳定运营：

1. 多租户或多部门隔离。
2. 权限控制和审计日志。
3. 批量审核。
4. 定时任务。
5. 审核质量评估集。
6. 模型效果对比和灰度发布。

## 9. 技术选型建议

在当前还未确定现有项目技术栈时，先给出通用建议：

| 模块 | 建议 |
| --- | --- |
| 后端 | Java Spring Boot 或 Python FastAPI，优先沿用现有项目 |
| 对外接口 | REST API 为主，必要时提供内部 SDK 或消息队列接入 |
| 文件解析 | Apache Tika、LibreOffice、pdfplumber、pymupdf、OCR 服务 |
| 向量库 | PostgreSQL + pgvector、Milvus、Qdrant |
| 全文检索 | PostgreSQL FTS、Elasticsearch、OpenSearch |
| 编排方式 | 先用后端状态机/任务表，后续再考虑 BPMN 或独立工作流引擎 |
| 异步任务 | Celery、Redis Queue、Spring Async、Quartz、消息队列 |
| AI 调用 | 封装统一 Model Gateway，屏蔽具体模型供应商差异 |

如果现有系统已经有后端框架、权限体系、文件服务和审批页面，建议工作流服务只做后端能力，不重复建设界面和用户体系。

核心表可以先按以下方向设计：

| 表名 | 作用 |
| --- | --- |
| audit_workflow | 工作流定义 |
| audit_workflow_node | 工作流节点配置 |
| audit_knowledge_base | 知识库定义 |
| audit_knowledge_document | 知识库来源文件 |
| audit_knowledge_chunk | 知识库切片 |
| audit_task | 审核任务 |
| audit_task_file | 待审核文件 |
| audit_task_chunk | 待审核文件切片 |
| audit_retrieval_record | 检索记录 |
| audit_result | 审核结果汇总 |
| audit_result_issue | 审核问题明细 |

## 11. 关键风险与应对

| 风险 | 应对 |
| --- | --- |
| 知识库检索不准 | 使用混合检索、元数据过滤、rerank 和人工标注评估集 |
| AI 输出格式不稳定 | 使用 JSON Schema 校验、失败重试、结构化输出约束 |
| 审核依据不可追溯 | 每条问题强制绑定 chunk_id、来源文件、章节或页码 |
| 文件解析质量差 | 针对 PDF、Word、Excel、扫描件分别建立解析策略 |
| 长文件超出上下文 | 分块审核、分层汇总、只传相关依据和局部内容 |
| 误报漏报 | 支持外部系统回写人工复核结论、建立历史样本回归测试、审核规则版本管理 |

## 12. 建议实施路线

### 阶段一：原型验证

目标：验证“文件解析 + 知识库检索 + AI 审核 + 结构化输出”是否可行。

交付物：

1. 一个固定审核工作流。
2. 一个小规模审核标准知识库。
3. 一个审核任务创建 API。
4. 一个审核任务状态查询 API。
5. 一份 JSON 格式审核结果。

### 阶段二：平台化

目标：支持三到四个审核工作流。

交付物：

1. 工作流配置表。
2. 知识库范围配置。
3. 提示词模板配置。
4. 输出 Schema 配置。
5. 审核任务状态机。
6. 异步回调或事件通知机制。

### 阶段三：运营化

目标：让审核流程可持续维护和评估。

交付物：

1. 人工复核结论回写接口。
2. 审核结果归档接口。
3. 统计分析。
4. 模型调用成本统计。
5. 知识库版本管理。

## 13. 当前待确认问题

后续讨论时建议优先确认以下问题：

1. 待审核文件主要是什么类型：PDF、Word、Excel、图片，还是混合格式？
2. 审核标准知识库是否已经存在，格式是什么？
3. 审核结果更偏向“问题清单”还是“完整审核报告”？
4. 是否要求离线部署或私有化模型？
5. 外部系统传入文件的方式是什么：文件 ID、文件 URL、对象存储地址、二进制流，还是直接传文本？
6. 三到四个工作流之间是共享知识库，还是各自绑定独立知识库？
7. 审核任务需要同步返回，还是异步回调？
8. 外部系统是否需要把人工复核结论回写给工作流服务？

## 14. 初步结论

建议先做轻量工作流服务，而不是独立建设前端平台，也不急于引入复杂 BPMN 引擎。第一版重点打通“系统调用 -> 文件解析 -> 知识库检索 -> AI 审核 -> 结构化结果返回”的闭环，并把工作流、知识库范围、提示词和输出结构配置化。这样既能快速落地第一个 AI 审核场景，也能支撑后续三到四个类似工作流扩展。
