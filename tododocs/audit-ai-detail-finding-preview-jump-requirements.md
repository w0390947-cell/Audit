# AI任务详情“检测结果点击跳转报告页”需求整理

日期：2026-05-12

## 1. 需求目标

在“AI任务详情”界面中，实现“检测结果”与“报告预览”的页码级联动。

用户在“检测结果”组件中点击某条 AI 检测问题后，“报告预览”组件自动跳转到该问题对应的报告页码，方便用户快速核对原文。

## 2. 页面范围

页面：

- `AI任务详情`

涉及组件：

- `报告预览`
- `检测结果`

## 3. 第一阶段范围

第一阶段只做“页码级跳转”：

1. 检测结果中的问题项可点击。
2. 点击问题后，读取该问题携带的页码字段。
3. 如果存在有效页码，报告预览跳转到对应页。
4. 如果没有页码，提示“暂无定位信息”。

第一阶段不做：

- 页内高亮；
- 坐标框选；
- 段落滚动；
- 原文选区；
- 多页同时定位；
- AI 结果与报告原文的二次匹配。

## 4. 推荐的报告预览实现方式

为了最简单、稳定地支持页码跳转，建议“报告预览”组件采用 PDF 分页预览方式。

原因：

- PDF 天然具备稳定页码。
- 页码跳转实现简单。
- doc/docx 的 HTML 预览分页不稳定，不适合作为页码定位基准。
- 后续如果需要高亮，也可以在 PDF 页基础上扩展坐标或文本定位。

建议方案：

1. 后端将用户上传的报告文件统一转换为 PDF 预览文件。
2. 前端“报告预览”加载 PDF。
3. 前端维护当前页码。
4. 检测结果点击后调用报告预览组件的 `jumpToPage(page)` 方法。

## 5. AI 结果数据要求

每条问题需要携带页码定位字段。

推荐结构：

```json
{
  "type": "数据错误",
  "title": "报告编号与依据文件不一致",
  "content": "报告首页编号与依据文件登记编号不一致。",
  "severity": "medium",
  "location": {
    "page": 1
  },
  "suggestion": "请核对并统一报告编号。"
}
```

前端后续可以兼容读取以下字段：

- `location.page`
- `location.pageNo`
- `location.page_no`
- `page`
- `pageNo`
- `page_no`

推荐工作流系统统一返回：

```json
{
  "location": {
    "page": 1
  }
}
```

## 6. 交互规则

### 6.1 有页码

当用户点击检测结果中的某条问题，且该问题存在有效页码：

1. 检测结果项进入选中状态。
2. 报告预览跳转到对应页。
3. 报告预览当前页码同步更新。

示例：

- 问题页码：`6`
- 用户点击问题
- 报告预览跳转到第 `6` 页

### 6.2 无页码

当用户点击的问题没有页码，或页码无效：

1. 不跳转报告预览。
2. 前端提示：

```text
暂无定位信息
```

无效页码包括：

- `null`
- 空字符串
- 非数字
- 小于 `1`
- 超出报告总页数

### 6.3 多页问题

第一阶段只跳转到一个默认页码。

如果后续工作流返回：

```json
{
  "location": {
    "page": 3,
    "pages": [3, 4]
  }
}
```

第一阶段只使用 `location.page = 3`。

## 7. 前端组件职责建议

### 7.1 检测结果组件

职责：

- 渲染问题列表。
- 将有页码的问题展示为可点击。
- 点击后提取页码。
- 无页码时提示“暂无定位信息”。
- 有页码时通知父组件或报告预览组件跳转。

建议方法：

```js
handleFindingClick(finding) {
  const page = this.resolveFindingPage(finding)
  if (!page) {
    this.$message.warning('暂无定位信息')
    return
  }
  this.$emit('jump-to-report-page', page)
}
```

### 7.2 报告预览组件

职责：

- 加载报告预览文件。
- 维护当前页码。
- 提供页码跳转方法。
- 页码无效时不执行跳转。

建议方法：

```js
jumpToPage(page) {
  // 校验 page
  // 更新 currentPage
  // 滚动或渲染到对应 PDF 页面
}
```

### 7.3 AI任务详情父组件

职责：

- 管理检测结果和报告预览之间的事件通信。
- 接收检测结果的跳转事件。
- 调用报告预览组件跳转。

建议结构：

```html
<report-preview ref="reportPreview" />
<finding-list @jump-to-report-page="handleJumpToReportPage" />
```

```js
handleJumpToReportPage(page) {
  this.$refs.reportPreview.jumpToPage(page)
}
```

## 8. 后端职责建议

第一阶段后端需要关注两件事：

1. 保存 AI 工作流返回的问题页码字段。
2. 提供稳定的 PDF 预览文件或可按页跳转的预览能力。

如果当前 `audit_ai_finding` 表没有页码字段，有两种方案：

### 方案 A：新增结构化字段

为 AI 问题表增加字段，例如：

- `page_no`
- `location_json`

优点：

- 查询和前端使用直接。
- 后续支持高亮或多页扩展更清晰。

### 方案 B：从原始结果 JSON 中解析

如果业务系统保存了工作流原始 result，可以前端或后端从原始 JSON 中解析页码。

缺点：

- 依赖原始 JSON 保存完整。
- 查询和展示逻辑更绕。
- 不利于后续扩展。

建议采用方案 A，至少保存 `page_no`，可选保存 `location_json`。

## 9. 验收标准

第一阶段验收标准：

1. 工作流返回带页码的问题后，业务系统能保存页码。
2. AI任务详情中，带页码的问题可以点击。
3. 点击问题后，报告预览跳转到对应页。
4. 点击没有页码的问题，提示“暂无定位信息”。
5. 页码从 `1` 开始，与报告预览页码一致。
6. 不影响原有检测结果展示。
7. 不影响原有报告预览加载。

## 10. 待确认问题

1. 工作流侧是否能返回 `location.page`？
2. 当前报告预览是否已有 PDF 转换产物；如果没有，需要确认后端转换方案。
3. `audit_ai_finding` 是否新增 `page_no` / `location_json` 字段。
4. 页码超出 PDF 总页数时，前端是否提示“暂无定位信息”还是“页码超出范围”。当前建议统一提示“暂无定位信息”。
